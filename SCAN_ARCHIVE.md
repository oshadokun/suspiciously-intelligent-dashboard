# Scan Archive Workflow

Every pipeline run is preserved in a dated folder under `/scans/`. Previous
runs are never overwritten. The dashboard stays fully offline; no APIs are
required for archival.

---

## Folder structure

```
/scans/
  scan_manifest.json        index of every archived run
  /latest/                  always the newest run — fast dashboard import
  /2026-05-06/              specific dated run
  /2026-05-08/              another run
  /2026-05-08_01/           second run on the same date (collision suffix)
  /2026-05-08_02/           third run on that date
```

Each dated folder contains:

| File | Description |
|---|---|
| `raw_research.json` | Raw stories from all sources |
| `analyzed_stories.json` | Scored and tagged stories |
| `content_ideas.json` | Generated content ideas for all formats |
| `data.json` | Dashboard-ready merged payload |
| `ideas.md` | Human-readable content ideas |
| `run_report.md` | Scan summary + archive metadata appended at bottom |
| `workspace_snapshot.json` | Workflow states and publish data (if present at archive time) |

---

## Archiving a new run

After every full pipeline run, execute:

```powershell
.\archive_scan.ps1
```

What it does, in order:

1. **Validates** all six required files are present — aborts if any are missing.
2. **Names** the archive folder `YYYY-MM-DD`. If that folder already exists, appends `_01`, `_02`, etc.
3. **Copies** all required files (plus `workspace_snapshot.json` if present) to the dated folder.
4. **Updates** `/scans/latest/` with the same files (overwriting the previous latest).
5. **Appends** an archive metadata block to `run_report.md`, then re-copies the updated report into the archive folder.
6. **Updates** `scan_manifest.json` with a new entry for this run.

### Options

```powershell
.\archive_scan.ps1                           # normal run
.\archive_scan.ps1 -DryRun                  # validate and report only, no writes
.\archive_scan.ps1 -SourceDir C:\other\path  # archive from a non-default directory
```

## Loading a scan into the dashboard

### Load the most recent scan

```powershell
.\load_latest_scan.ps1
```

Copies `/scans/latest/` into the project root, then open `dashboard.html`.

### Load a specific historical scan

```powershell
.\load_latest_scan.ps1 -ScanDate 2026-05-06
.\load_latest_scan.ps1 -ScanDate 2026-05-08_01
```

### Browse available scans without loading anything

```powershell
.\load_latest_scan.ps1 -ListOnly
```

Prints the full manifest with story counts and top stories.

### Manual dashboard import

Alternatively, use the dashboard's built-in **Import new scan** button:

1. Click **Import new scan** in the dashboard header.
2. For **data.json**, browse to `scans/2026-05-06/data.json`.
3. For **content_ideas.json**, browse to `scans/2026-05-06/content_ideas.json`.

This does not require running `load_latest_scan.ps1` and does not modify any
files on disk.

## Collision handling

If the pipeline runs more than once on the same calendar date:

```
First run today   →  scans/2026-05-08/        (no suffix)
Second run today  →  scans/2026-05-08_01/
Third run today   →  scans/2026-05-08_02/
```

`/scans/latest/` is always updated to whichever run was archived most recently,
regardless of suffix.

## scan_manifest.json

A running JSON array written to `/scans/scan_manifest.json`. Each entry:

```json
{
  "scan_id":        "scan-2026-05-06",
  "date":           "2026-05-06T12:00:00Z",
  "folder":         "2026-05-06",
  "story_count":    22,
  "verified_count": 9,
  "high_risk_count": 5,
  "top_cluster":    "regulation",
  "top_story":      "Apple agrees to pay $250 million..."
}
```

| Field | Source |
|---|---|
| `scan_id` | `scan-{folder}` |
| `date` | `data.json → meta.scan_date`, else archive timestamp |
| `folder` | Dated folder name |
| `story_count` | `data.json → stories.length` |
| `verified_count` | Stories where `verified: true` |
| `high_risk_count` | Stories where `risk_level: "high"` |
| `top_cluster` | Most common `source_category` in `analyzed_stories.json` |
| `top_story` | Highest-scoring story headline in `data.json` |

New entries are appended; existing entries are never modified.

## run_report.md archive block

After archiving, the following block is appended to the live `run_report.md`
(and re-copied into the dated archive folder):

```markdown
---

## Archive metadata

| Field | Value |
|---|---|
| Scan timestamp | 2026-05-08T14:30:00Z |
| Archive folder | scans/2026-05-08 |
| Latest updated | Yes |
| Total archived scans | 2 |
| Scan ID | scan-2026-05-08 |
```

## Workspace snapshot

`workspace_snapshot.json` is not generated by the pipeline automatically.
Export it manually from the dashboard before archiving:

1. Click **Export workspace** in the dashboard header.
2. Save the file as `workspace_snapshot.json` in the project root.
3. Run `.\archive_scan.ps1` — the snapshot will be included automatically.

## Known limitations

| Limitation | Notes |
|---|---|
| Manual trigger | `archive_scan.ps1` must be run explicitly after each pipeline run. There is no automatic post-run hook yet. |
| No automatic cleanup | All scan folders are kept indefinitely. Delete old dated folders manually if disk space is a concern. |
| Top-cluster approximation | `top_cluster` in the manifest uses the most common `source_category` field, not the narrative clustering algorithm the dashboard uses. |
| Windows PowerShell only | Scripts require PowerShell 5.1+. They run on Windows out of the box; macOS/Linux requires PowerShell 7+. |
| No checksums | Files are copied by name; there is no hash verification of archived content. |
